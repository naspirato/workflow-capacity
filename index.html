Монолит vs Sharding — отличия

Два режима исполнения PR-check relwithdebinfo, которые сравнивает симулятор.

Монолит
  • 1 VM на весь PR-check job
  • Меньший burst по пулу — проще вписаться в квоты
  • Оптимален для лёгких PR (D < 60 мин)
  • Wait обычно ниже при том же пуле
  • Work = полное время компиляции/тестов на одной машине
PR-check sharding
  • Prepare VM + N параллельных shard VM
  • Сильно сокращает work на тяжёлых PR (D ≥ 60 мин)
  • Требует N+1 раннеров одновременно → выше риск очереди
  • ~8% overhead на фазу тестов (SHARD_OVERHEAD 1.08)
  • На пике UTC shard cap = 4 независимо от tier

Когда sharding выигрывает по total time

Группа D Shard count Типичный эффект Условие выигрыша
D < 60 мин 1 Без sharding Монолит ≈ sharding
60 ≤ D < 120 мин 4 −20…−40% work Квота позволяет 4+1 раннеров без длинной очереди
120 ≤ D < 200 мин 8 (peak: 4) −35…−55% work Выигрыш растёт, но wait растёт при узком пуле
D ≥ 200 мин 12 (peak: 4) −50…−65% work Максимальный профит wall-clock; критичен VM budget

Проценты work — ориентир по модели (параллель тестов / overhead), не замена симуляции. Реальные p90 wait/work/total — из прогона ноутбука на кэшированной истории jobs.

Когда увеличивать квоты — чеклист

Квоты помогут, если

  • Высокий wait p90 в пиковые часы (09–15 UTC)
  • Sharding быстрее по work, но delta_total > 0 из-за очереди
  • capacity_cap режет желаемое число шардов
  • Много PR в группах D ≥ 60 мин
  • Пул часто saturated (saturated_seconds высокий в симуляции)

Квоты не решат, если

  • Узкое место — compile phase, а не test/shard
  • Большинство PR лёгкие (D < 60) — sharding не включается
  • Peak cap 4 shard ограничивает выигрыш в рабочие часы
  • Проблема в classifier / eligibility, а не в ёмкости пула

Метрики сравнения (из симулятора)

Метрика Что значит Монолит Sharding
wait p90 Время в очереди до выделения раннера Обычно ниже Выше при burst N+1 VM
work p90 Активное время на раннере Полный job на 1 VM Сжато за счёт параллели
